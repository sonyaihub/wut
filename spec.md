# wut — Spec

## 1. Problem

A user opens a new terminal and, out of habit, starts typing natural language at the prompt (e.g. `how do I rebase this branch onto main`) as if chatting with an AI harness. The shell responds with `command not found`, which is noisy and wastes the intent. We want to catch these moments and hand the typed text off to the user's configured AI harness (Claude Code, codex, opencode, custom CLI, etc.) as a prompt.

## 2. Goals

- Detect natural-language input at an interactive shell prompt with near-zero false positives on real commands.
- Route the detected text to a user-configured harness with the text as its initial prompt.
- Support both **interactive** mode (hand the tty to the harness) and **headless** mode (one-shot, print answer inline, shell stays in control) per harness.
- Ship presets for common harnesses (Claude Code, codex, opencode, etc.) so 90% of users never write invocation flags by hand.
- Be install-once, opt-in, and trivially removable.
- Zero measurable latency cost on normal shell usage (no wrapper around every command).

## 3. Non-goals (v1)

- Not a shell replacement, REPL, or multiplexer.
- No inline autocomplete, no "did you mean" suggestions for real commands.
- No harness management (installing Claude Code, opencode, etc. is the user's job).
- No history or session persistence beyond what the harness itself does.
- Bash/fish support — zsh only for v1.

## 4. User experience

### Install
```
brew install wut             # or: curl | sh, or: go install
wut init zsh >> ~/.zshrc     # prints a shell snippet to source
source ~/.zshrc
wut setup                    # guided: pick harness, pick default mode
```

`setup` is interactive:
```
? Which harness do you use? (↑/↓)
  > claude          (Claude Code — detected on PATH)
    codex
    opencode
    cursor-agent
    custom...
? Default mode when natural language is detected?
    interactive    (opens the harness, replaces the terminal)
  > headless       (prints a one-shot answer, stay in shell)
    ask            (prompt me each time)
? Headless output style?
  > stream         (tokens print as they arrive)
    buffered       (render after completion)
✔ wrote ~/.config/wut/config.toml
✔ run `wut doctor` to verify
```

Non-interactive shorthand:
```
wut setup --harness claude --mode headless
```

### Happy path — interactive mode
```
$ how do I rebase onto main without losing my stash
→ launching claude with: "how do I rebase onto main without losing my stash"
[claude opens, prompt pre-filled, terminal now belongs to claude]
```

### Happy path — headless mode
```
$ how do I rebase onto main without losing my stash
⠋ claude is thinking…            ← spinner shown until first byte of output

╭─ claude ────────────────────────────────────────────╮
│ Stash your work first with `git stash`, then        │
│ `git rebase origin/main`. After the rebase,         │
│ `git stash pop` to re-apply. If there are           │
│ conflicts during pop, resolve and `git add`...      │
╰──────────────────────────────────────────────────────╯
$                        ← shell prompt returns, no mode switch
```
Output is streamed (or buffered) into the scrollback; the user stays at their shell. Good for quick one-shot questions. Bad for long sessions — for those, interactive wins.

**Spinner:** a one-line spinner (`⠋⠙⠹⠸⠼⠴⠦⠧⠇⠏`, ~100ms frame) with the label `<harness> is thinking…` is drawn on the line the shell returned. It's erased the moment the first byte of stdout arrives (streaming) or the process exits (buffered). The spinner line uses `\r` + clear-to-end-of-line so it never leaves a ghost in scrollback. Disabled automatically when stdout is not a tty (e.g. piped), and suppressible with `behavior.spinner = false`.

### Ask mode
```
$ how do I rebase onto main
? open claude…
  > headless      (one-shot answer here)
    interactive   (hand over the terminal)
    cancel
```

### Negative path (real command, unchanged)
```
$ ls -la
(runs ls normally)

$ gti status
zsh: command not found: gti
```
Single-token typos still fall through to the shell's normal error. We only intercept things that *look* like natural language.

### Confirm mode (configurable, default off)
```
$ how do I rebase onto main
→ open claude with this prompt? [Y/n]
```

## 5. Architecture

Two pieces:

1. **`wut` binary** (Go, single static binary).
    - Subcommands: `init`, `config`, `detect`, `run`, `doctor`.
    - Owns config parsing, detection heuristic, harness launching.
2. **Shell snippet** (generated by `wut init zsh`).
    - Defines a zsh `command_not_found_handler` function.
    - Forwards the raw input line to `wut detect --line "$*"`.
    - Exit code of `detect` determines behavior:
      - `127` → not natural language; let zsh fall through to its default "command not found" message.
      - Any other code → handled. The snippet propagates it so the shell sees the harness's actual exit code (including non-zero from a crashed harness). This matters because interactive mode replaces the process via `exec`, so `detect` can't control its own exit code once the harness has taken over.

Why `command_not_found_handler` and not `preexec`:
- `preexec` runs before *every* command. Intercepting there means we'd need to re-parse and re-decide for every `ls`, `git`, etc. Higher latency, higher risk of breaking real commands.
- `command_not_found_handler` only fires when the shell has already determined the first token is not a command, function, alias, or builtin. That's a strong prior for "this isn't a shell command."

## 6. Detection heuristic

Input arrives as an array of tokens (zsh splits on whitespace). The heuristic must be cheap and explainable.

A line is classified as **natural language** if it passes all hard gates and at least two soft signals.

### Hard gates (all must be true)
- `argc >= 3` — single-word and two-word inputs are almost always typos (`gti`, `pythno script.py`).
- First token is not an existing command, alias, function, or builtin (guaranteed by `command_not_found_handler`).
- First token contains no `/`, no `.`, no `-`, no `~`, no `$`, no env-var prefix, no leading `./`. These indicate path/flag-style usage (a typo'd path, not English).
- No shell metacharacters outside quoted strings: `|`, `>`, `<`, `&`, `;`, `$(`, backticks. If any are present, treat as shell intent and pass through.

### Soft signals (need ≥ 2)
- Contains at least one common English stopword (`the`, `a`, `is`, `how`, `what`, `why`, `can`, `I`, `my`, `to`, `do`, `does`, `should`) — case-insensitive, whole-word match.
- Line ends with `?` or contains `?` mid-line.
- Contains a contraction or punctuation typical of prose (`'m`, `'s`, `n't`, `,`).
- `argc >= 6` (long lines are rarely mistyped commands).
- First token is a common interrogative or imperative (`how`, `what`, `why`, `explain`, `write`, `make`, `fix`, `help`).

### Explicit escape hatch
If the line starts with `\` or `?`, skip the heuristic entirely and always route to the harness. Example: `? what does this do` always opens the harness. This lets a user deliberately invoke even with short input.

### Explicit opt-out
If the line starts with `!`, never route — always fall through. (`!gti` behaves exactly like `gti`.)

## 7. Configuration

File: `~/.config/wut/config.toml`

The core model: users pick a **harness** (a named entry with interactive + optional headless invocation) and a **default mode**. Each mode describes how to invoke the harness.

```toml
# Which harness to use and how to run it by default.
active_harness = "claude"
default_mode   = "headless"    # "interactive" | "headless" | "ask"

[behavior]
confirm       = false          # Y/n prompt before launching
escape_hatch  = true           # `?` / `\` prefix always routes
passthrough   = ["gti", "sl", "pythno"]
spinner       = true           # show "<harness> is thinking…" spinner in headless mode
spinner_style = "dots"         # "dots" | "line" | "pipe" | "none"

[detection]
extra_stopwords = []

# -----------------------------------------------------------------------------
# Harness definitions. Ship as presets; users can override or add their own.
# -----------------------------------------------------------------------------

[harness.claude]
# Interactive: hand the tty over. {prompt} is substituted; if absent, piped on stdin.
interactive = { command = "claude", args = ["{prompt}"] }

# Headless: one-shot. Stdout is captured and rendered inline.
# `stream = true` means we render tokens as they arrive (line-buffered).
headless = {
  command = "claude",
  args    = ["-p", "{prompt}"],
  stream  = true,
}

[harness.opencode]
interactive = { command = "opencode", args = ["{prompt}"] }
headless    = { command = "opencode", args = ["run", "{prompt}"], stream = true }

# Example of an interactive-only custom harness: leaving `headless` unset
# means "headless requests auto-fall-back to interactive for this harness".
# Users can override this with behavior.headless_fallback = "error".
[harness.interactive-only-example]
interactive = { command = "my-tui-agent", args = ["--message", "{prompt}"] }

[harness.codex]
interactive = { command = "codex", args = ["{prompt}"] }
headless    = { command = "codex", args = ["exec", "{prompt}"], stream = true }

[harness.custom]
# Fully user-defined example.
interactive = { command = "my-agent", args = ["--tui"], stdin = "{prompt}" }
headless    = { command = "my-agent", args = ["--once", "--prompt", "{prompt}"] }
```

### Invocation spec (shared by interactive and headless)

Each mode block accepts:

| Field | Type | Meaning |
|---|---|---|
| `command` | string | Executable name or path. Must be resolvable via `$PATH` or absolute. |
| `args` | `[]string` | Argument list. `{prompt}` is substituted wherever it appears (supports multiple occurrences). |
| `stdin` | string? | If set, its value (with `{prompt}` substituted) is piped to stdin. Useful for harnesses that take the prompt via stdin instead of argv. |
| `env` | `map<string,string>`? | Extra env vars for the child, `{prompt}` substitution allowed. |
| `stream` | bool (headless only) | Line-buffer stdout into the terminal live. Default `true`. |
| `render` | `"raw" \| "box" \| "markdown"` (headless only) | How to wrap the output. Default `"box"` for visual separation from shell prompts. |
| `timeout_sec` | int? (headless only) | Kill the harness if it exceeds this. Default unlimited. |

### Headless-specific behavior

- **Stdout** is captured and rendered inline. Tokens stream by default.
- **Stderr** from the harness is forwarded to the user's stderr.
- **Exit code** propagates; non-zero renders a red indicator under the output box.
- **Interrupts** — Ctrl-C during a headless run sends SIGINT to the child and returns to the shell.
- **No tty** — we explicitly don't allocate a pty, so harnesses that require a tty for their "real" experience won't work headless. Those should declare `headless` unset; we fall back per policy.

### Fallback policy

Controlled by `behavior.headless_fallback`:

| Value | When headless is requested but no `headless` block exists |
|---|---|
| `interactive` (default) | Silently use the harness's interactive mode. |
| `ask` | Prompt the user to choose. |
| `error` | Print an error and exit 127 (line falls through to shell). |

### Ergonomic CLI

```
wut setup                          # guided
wut harness list                   # show configured harnesses + presets
wut harness use claude             # set active_harness
wut harness test claude --headless "say hi in one word"
wut mode set headless              # set default_mode
wut config path | edit | get | set
```

`harness test` is important — it runs the invocation without going through detection, so users can validate their config without hunting for a natural-language phrase the heuristic will pick up.

## 8. Subcommand surface

| Command | Purpose |
|---|---|
| `wut init zsh` | Print shell snippet to stdout (for `>>` into rc file). |
| `wut setup` | Guided config wizard: pick harness, default mode, headless output style. |
| `wut detect --line "<text>"` | Classify and act. Used by the shell hook. Exit 0 = handled, 127 = pass through. |
| `wut run --line "<text>" [--mode=...]` | Skip detection, always launch the harness. `--mode` overrides `default_mode`. Exposed for key-bindings and scripts. |
| `wut harness list\|use\|test\|add` | Manage harnesses. `test` invokes without detection so users can verify a config. |
| `wut mode set <interactive\|headless\|ask>` | Shortcut for `default_mode`. |
| `wut config get\|set\|path\|edit` | Config management. |
| `wut doctor` | Verify: config valid, active harness binary in PATH, both modes invokable (if configured), shell hook installed, version. |
| `wut version` | Print version. |

## 9. Harness launch semantics

Resolve the invocation block by `(active_harness, effective_mode)` where `effective_mode` = `--mode` flag ?? `default_mode` ?? `interactive`.

### Interactive mode
1. Substitute `{prompt}` in every arg and in `stdin`/`env` values.
2. `exec` the harness in the foreground, inheriting stdin/stdout/stderr/tty. The user ends up inside the harness; exiting returns to the shell.
3. Propagate the harness's exit code.
4. Never daemonize. Never background.

### Headless mode
1. Substitute `{prompt}` as above.
2. Spawn the harness as a child. **Do not** allocate a pty. Connect stdin from the configured `stdin` value (or `/dev/null`), capture stdout, forward stderr directly to the user's stderr.
3. If `behavior.spinner` is true and stdout is a tty, start a spinner on the current line with label `<harness> is thinking…`. Erase it (write `\r\x1b[K`) on the first byte of stdout or on process exit, whichever comes first. The spinner is purely cosmetic — never blocks I/O, never buffers harness output.
4. Render stdout per `render` setting:
   - `raw`: bytes straight to terminal (line-buffered if `stream = true`).
   - `box`: wrap in a Unicode box with harness name as a label. Box redraws as content streams (write header, stream lines, close footer on EOF).
   - `markdown`: lightweight markdown-to-ANSI rendering (headings, code fences, bold/italic, lists). Only applied in buffered mode; streaming markdown renders as `raw`.
5. Propagate exit code. On non-zero, render a red ✗ indicator below the output.
6. Ctrl-C sends SIGINT to the child; after two SIGINTs within 500ms, we SIGKILL and return to the shell. The spinner (if still running) is erased first.
7. Enforce `timeout_sec` if set; on timeout, SIGTERM then SIGKILL after 2s grace.
8. Return the user to their shell prompt.

## 10. Shell snippet (illustrative)

```zsh
# installed by `wut init zsh`
command_not_found_handler() {
  # guard against recursion if wut isn't (or is no longer) on PATH
  if ! command -v wut >/dev/null 2>&1; then
    print -u2 "zsh: command not found: $1"
    return 127
  fi

  wut detect --line "$*"
  local rc=$?
  # 127 = classifier said pass through. Anything else = handled (the exit
  # code is whatever the exec'd harness returned).
  if [ $rc -eq 127 ]; then
    print -u2 "zsh: command not found: $1"
    return 127
  fi
  return $rc
}
```

The binary handles all logic; the snippet is intentionally dumb so users can eyeball it. The `command -v` guard matters because a hook installed into `.zshrc` can otherwise recurse infinitely if the binary falls off `PATH`.

## 11. Edge cases

| Case | Behavior |
|---|---|
| Empty line | Never invoked — shell doesn't call the handler. |
| Piped input `how do I | grep foo` | Shell metacharacter gate trips → pass through. |
| Huge paste (multi-line) | zsh calls handler per line; we classify each. If user pastes code, first line usually has symbols and fails the gates. |
| Harness binary missing | `detect` errors with actionable message ("configure harness.command or install claude"), exit 127 so the user still sees a command-not-found-like result. |
| SSH session | Works the same — binary just needs to be on the remote. |
| Non-interactive shell (scripts) | `command_not_found_handler` still fires in scripts. We detect `$-` for interactive-only and skip if not interactive. |
| User is inside tmux / screen | No special handling needed. |
| User has their own `command_not_found_handler` already | `init zsh` detects this and refuses to overwrite; prints instructions to chain manually. |

## 12. Security & privacy

- The detected line is passed only to the user-configured local binary. No network calls from `wut` itself.
- Detection runs locally; no telemetry by default. A `--telemetry` flag can be added later if we want opt-in metrics.
- Because a typo like `rm -rf /tmp/foo` would pass the metachar gate (no metachars) but fail hard gates (contains `/`, `-`), it won't be hijacked. Still worth calling out in a threat-model section that we never *execute* the detected text — we only pass it as a string argument to the harness.

## 13. Open questions

1. **Multi-harness profiles.** Should a user be able to define multiple harnesses and pick based on prefix (`@c ...` → claude, `@o ...` → opencode)? Probably yes in v2.
1a. **Per-prompt mode prefixes.** Even with a default mode, users will want a quick override. Proposal: `??` prefix forces headless, `?!` forces interactive. Cheap to add and composes with the existing `?` / `!` escape hatches. Defer unless it comes up in dogfooding.
1b. **Mode promotion mid-flight.** Headless answer is unsatisfying and the user wants to "continue the conversation." Should there be a hotkey at the end of a headless render to re-launch the harness interactively with the same prompt as seed? Nice-to-have; out of scope for v1.
2. **Paste detection.** If a user pastes a long multi-line block, we currently classify each line. Should we debounce / buffer? Out of scope for v1 until we see real misses.
3. **Learning.** Should `passthrough_tokens` auto-update when a user repeatedly escapes with `!`? Tempting but adds state. Defer.
4. **Cross-platform.** Binary is cross-platform; the shell hook is zsh-only for v1. Bash (`command_not_found_handle`, note spelling) and fish (`fish_command_not_found`) are straightforward follow-ups.

## 14. Milestones

- **M0 — Walking skeleton.** `detect` with heuristic, `init zsh`, hardcoded harness = `echo`. Proves the hook path.
- **M1 — Interactive launch.** TOML config, harness presets (claude/aider/codex), `{prompt}` substitution, `exec` semantics, `doctor`, `harness test`.
- **M2 — Headless mode.** Streaming stdout, `box` renderer, interrupt handling, timeout, fallback policy. Default mode becomes configurable.
- **M3 — Setup UX.** `wut setup` wizard, `ask` mode, per-invocation `--mode` override, `markdown` renderer.
- **M4 — Polish.** `confirm` mode, escape hatch, passthrough tokens, error messages, `harness add`.
- **M5 — Distribution.** Homebrew tap, install script, `init bash` / `init fish`.

## 15. Success criteria

- On a test corpus of 100 real commands + 100 natural-language prompts, classifier achieves ≥ 99% precision on commands (i.e. ≤ 1 command misrouted to harness) and ≥ 90% recall on natural language.
- Zero measurable latency (`hyperfine` diff) for normal command execution, since we only run on `command_not_found`.
- A user can install, configure, and experience the happy path in under 60 seconds.
